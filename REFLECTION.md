# Reflection

### 1. The hardest bug you hit this week, and how you debugged it
The hardest bug I encountered this week was related to hydration mismatches and UI clipping when integrating Base UI primitives with the Next.js App Router. Specifically, the dynamic tool selector dropdown on the `/audit` page was clipping behind other fixed elements on mobile screens, and attempting to fix it using the `asChild` prop resulted in severe React hydration errors because the server-rendered DOM did not perfectly match the client-rendered DOM. To debug this, I first isolated the component outside of the complex form to verify the core issue. I realized that the `asChild` prop was passing incompatible attributes to a nested native HTML element. I reversed course, dropped the `asChild` pattern for this specific component, and instead utilized a React Portal to render the dropdown at the root of the DOM. This completely bypassed the z-index stacking context issues that were causing the clipping, and removing the forced prop-drilling eliminated the hydration mismatch, resulting in a smooth, responsive UI.

### 2. A decision you reversed mid-week, and what made you reverse it
Mid-week, I completely reversed my decision on how to calculate the user's financial savings. Initially, my plan was to send the raw user input (team size, tools, spend) to the Google Gemini API and prompt the LLM to mathematically calculate the "ideal" spend and return the savings. However, after early prototyping, I realized that LLMs are fundamentally unreliable at deterministic math. The API would occasionally hallucinate completely incorrect plan prices or botch basic multiplication, which is unacceptable for a financial audit tool. I reversed this decision and built a strict, pure-function, deterministic rules engine (`lib/audit/engine.ts`) in TypeScript. The engine handles all the math and overlapping tool logic natively using hardcoded pricing arrays. I then passed only the *final, calculated results* to the LLM to generate a qualitative text summary. This architectural shift saved the product's credibility.

### 3. What you would build in week 2 if you had it
If I had a second week to work on SpendLens AI, my primary focus would be drastically reducing friction at the top of the funnel. Currently, users must manually input every AI tool they use, select the tier, and estimate their monthly spend. In week 2, I would integrate the Plaid API or Stripe API to allow founders to securely connect their bank accounts or corporate cards. The application would automatically scan their recent transactions for AI vendor footprints (e.g., Anthropic, OpenAI, GitHub), parse the spend, and auto-populate the audit engine. Additionally, I would build an internal Admin Dashboard for the Credex sales team. This dashboard would automatically rank incoming leads based on their "Efficiency Score," allowing the sales team to prioritize outreach to companies that are bleeding the most cash on redundant tools.

### 4. How you used AI tools, what you did not trust them with, and one time AI was wrong
I used AI tools (specifically GitHub Copilot and ChatGPT) heavily for scaffolding boilerplate code, such as writing the initial Drizzle ORM schema definitions and generating the base structure for my Vitest test cases. However, I explicitly *did not trust* AI tools with the core business logic or the financial calculations. As noted earlier, relying on an LLM to dynamically calculate enterprise software savings proved disastrous. One specific time AI was wrong occurred when I asked Copilot to write a regex to validate the complex pricing tiers. It generated a overly restrictive regex that failed to account for variable "Pay-as-you-go" API pricing, which would have broken the form for any backend engineering team. I had to manually rewrite the validation logic using simple, robust TypeScript type guards instead.
