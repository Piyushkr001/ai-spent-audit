# Reflection

### 1. What is the most important thing you learned during this project?
The most important thing I learned is the immense value of separating deterministic business logic from probabilistic AI generation. Initially, I considered using an LLM to calculate the financial savings based on the user's tool input. However, I realized that LLMs are notoriously unreliable with exact math and consistency. By building a strict, pure-function deterministic engine to handle the pricing matrix and savings rules, I ensured that the core value proposition of the app—the financial math—is 100% accurate, testable, and reliable. I then scoped the AI's role strictly to generating qualitative executive summaries. This hybrid architecture (deterministic math + probabilistic text) is a pattern I will carry forward into all future enterprise AI applications.

### 2. What was the most difficult technical challenge you faced?
The most difficult technical challenge was managing the state and validation of the dynamic tool input form on the `/audit` page. Users can add an arbitrary number of tools, change the plans for each tool, and adjust the seat count. The pricing logic dictates that some plans are fixed per seat, while others (like APIs or Enterprise tiers) are variable or quote-based. Syncing this complex, deeply nested UI state with a validation layer that instantly feeds into the deterministic engine without causing UI stuttering or re-render loops required careful memoization and state structure planning. Additionally, resolving hydration mismatches when integrating Base UI primitives into the Next.js App Router took considerable debugging time.

### 3. If you had 2 more weeks to work on this, what would you add or change?
If I had two more weeks, I would implement OAuth authentication (Google/GitHub) so users could log in to view a dashboard of their past audits, rather than relying solely on the emailed link to the public report. I would also integrate Plaid or Stripe APIs to allow users to automatically pull their actual software subscriptions from their bank statements, completely removing the manual data entry step. Finally, I would build an internal Admin Dashboard for the Credex sales team, allowing them to view incoming high-value leads, sort them by efficiency score, and directly reach out to founders who are bleeding cash on redundant AI tools.

### 4. How did you decide on your technical stack?
I chose Next.js (App Router) for its seamless blend of server-side execution and client-side interactivity, which was perfect for hiding my database and email API keys while providing a fast UI. I selected Drizzle ORM and Neon Postgres because Drizzle provides unmatched end-to-end type safety, and Neon's serverless architecture means I don't pay for idle database time. I opted for Tailwind CSS and Shadcn UI to rapidly build a beautiful, accessible interface without writing custom CSS classes from scratch. Finally, I chose Google Gemini over OpenAI simply due to the generous free tier limits during development, ensuring I wouldn't hit rate limits while testing the summary generation.

### 5. How did the Credex requirements influence your product decisions?
The primary goal was to generate high-quality consultation leads for Credex. This directly influenced my decision to implement a deterministic engine that aggressively searches for overlaps (e.g., paying for both Cursor and Copilot) and enterprise waste. The engine is explicitly tuned to surface high savings. I also deliberately chose *not* to gate the initial audit results behind an email wall. Instead, the user gets immediate value (seeing their savings on the screen) and is only asked for their email when they want to save the report or share it with their team. This "give value before extracting value" product decision directly increases trust and, theoretically, the conversion rate for high-value Credex leads.
